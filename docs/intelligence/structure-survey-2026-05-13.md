# Structure Survey — 7 Candidates for AngelEye's Structure Layer

## Premise

AngelEye's planned **tag layer** answers "what facts exist about this session?" — a BMAD orchestrator, a Paperclip workspace, a delamain worker, a git outcome. Tags POINT AT richer concepts but they don't encapsulate them. A BMAD lifecycle has phases, transitions, and outcomes; a Ralphy campaign has waves and a plan; a Playwright UAT run has assertions and routes covered. **Structures model what those concepts ARE.**

David's instruction: design from one pattern fails. Survey 6–7 candidates first, then look at the data to see the shared vs divergent slice.

This doc is the survey. It does NOT propose a schema (that's a follow-up). It does NOT redesign the tag layer (the unified-tag-architecture spec covers that). It DOES read the existing synthesis docs and EXTEND them with structure-layer thinking.

The 7 candidates surveyed:

1. BMAD lifecycle
2. Ralphy campaign
3. Ruflo swarm
4. Playwright interaction (UAT / screenshot tour / investigation / visual)
5. ALS delamain batch
6. Paperclip hosting
7. AppyCtrl probe sequence

## 1. BMAD Lifecycle

**Codified intent**: `/appydave:bmad-story-lifecycle` orchestrator drives ~9 phases per story (Bob CS → Bob VS gate → Amelia DS → Nate DR → Taylor CS → Taylor RA → Lisa CU → Swagger self-maintenance → Ship). Two harness modes seen in corpus: **Mode 2** (separate iTerm2 panes per agent — broken in Claude Code 2.1.212) and **Mode 1** (Agent Teams in-process via TeamCreate + Agent + SendMessage — current default). Mode 3 (in-context fallback) is a silent failure mode.

| Aspect                   | Detail                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --- | --- | --- | --- | ---- | ----------------------------------------------------------------------------------------------------------------------------------- |
| **Scope**                | **Hybrid.** Pre-2026-04-06: cross-session (1 orchestrator + ~9 phase sessions). Post-2026-04-06: within-session (1 orchestrator session with N in-process Agent calls). Same conceptual lifecycle, two physical realisations.                                                                                                                                                                                                                                                                                                                                                                                                    |
| **Lifecycle model**      | **Phases (named, ordered, gated).** Canonical 9 phases plus optional Amelia/Nate fix loops with incrementing suffix (`amelia-fix-1`, `amelia-fix-2`). Two human gates (Bob VS, ship).                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| **Constituent sessions** | Mode 1: 1 orchestrator session; phases are `Agent` tool calls inside it. Mode 2: 1 orchestrator + 9–11 phase sessions, sequential by intent but timestamps show overlap when fix loops kick in. 72% of Mode-1 runs land at 7–9 Agent calls; 20% at 10–12 (one fix loop); 3% at 13+.                                                                                                                                                                                                                                                                                                                                              |
| **Aggregation key**      | **`trigger_arguments` carries `story_id`** (e.g. `0.63`, `0.64`, `5.1`). Visible at session_start. Plus session `name` field on Mode-2 phase sessions encodes story id (`bmad-sm CS 5.1`). **Not extracted into a typed field today** — lives inside `trigger_arguments` text.                                                                                                                                                                                                                                                                                                                                                   |
| **Outcomes**             | Story shipped (ship phase completed + git commit landed). Or: lifecycle aborted/restarted (~5 stories invoked 2–3× same day; no resume semantic, restart = full new orchestration). Mode-3 silent fallback (Swagger embodied agents instead of spawning them — 14% of orchestrator runs) is a codified-contract violation that succeeds but should be flagged.                                                                                                                                                                                                                                                                   |
| **Linkage signals**      | Mode 1: all phase work inside one `session_id` — linkage is trivial (same session). Mode 2: `trigger_command =~ /^bmad-(sm                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       | dev | dr  | sat | lib | ship | oversight)/ AND trigger_arguments contains <story_id>`. Same-project + same-day + same-story-id is deterministic enough to cluster. |
| **Internal anatomy**     | Sequential pipeline with conditional fix-loops at Nate DR gate. Each phase produces a deterministic artifact (story file edits, code, tests, SAT report, curation entry, ship commit). Mode 1 internal anatomy is observable via `TeamCreate=1 + Agent=N + SendMessage=N-1` per orchestrator session.                                                                                                                                                                                                                                                                                                                            |
| **Corpus evidence**      | **383 BMAD-keyword sessions**, 84 orchestrators, 267 phase agents. SupportSignal is 96% of all BMAD sessions. Sample corpus IDs (probed 2026-05-13): `46cd9d04` (story 0.63, Mode 1, 2026-05-12), `90380a4f` (0.64), `829d9724` (0.65), `0bb9b136` (0.66), `df952bd8` (0.63 re-run). All `trigger_command: appydave:bmad-story-lifecycle`, all `subtype_heuristic: build.bmad_orchestrator`, all `subtype: build.bmad_orchestrator` post-2026-05-06 (classifier fix). Pre-fix: 79/84 still tagged `build.campaign`. Mode-1 sample `ba239633` (story 0.48, `TeamCreate=1, Agent=7, SendMessage=5, TaskCreate=15, TaskUpdate=26`). |
| **Open questions**       | (a) Should the structure track BOTH Mode 1 and Mode 2 the same way, or are they two structure-variants? (b) Story-id extraction to a typed field — should it live on the structure or get backfilled onto sessions? (c) Phase identity (CS/VS/DS/DR/CU) is in `name` and `first_prompt`; not a structured field. (d) How does the structure represent the human gates — as transitions, not stations? (e) Re-runs of the same story_id: separate structures or one with attempts[]?                                                                                                                                              |

**Verdict: needs a structure.** Strong case — repeated, gated, named-phase lifecycle, observable across the corpus. Both within-session (Mode 1) and cross-session (Mode 2) variants must be modelled.

## 2. Ralphy Campaign

**Codified intent**: `/appydave:ralphy` — David's "Ralph Wiggum" approach for autonomous batch campaigns. Four modes (Requirements / Plan / Build / Extend). Three profiles (Development 3-5 agents, Analysis 8-9 agents, Content 6 agents) each with their own `AGENTS-{profile}.md`. Lifecycle: Planning → Execution → Validation → Assessment. "campaign = PR = worktree" interchangeable. Mandatory bold workflow gates in every response. Mandatory `appydave:delivery-review` between waves.

| Aspect                   | Detail                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Scope**                | **Typically within-session** (1 huge coordinator loop). Can span sessions when extended via handover (Mode 4). Each campaign is "campaign = PR = worktree" — physical 1:1 mapping.                                                                                                                                                                                                                                                                            |
| **Lifecycle model**      | **Waves (numbered, sequential, gated).** Each wave has a fixed agent count (3-5 Dev, 8-9 Analysis, 6 Content). Bracketed by delivery-review at wave boundaries and campaign end. AGENTS.md inherited wave-to-wave. BACKLOG.md (B### IDs) is the persistent record.                                                                                                                                                                                            |
| **Constituent sessions** | Usually 1 (the marathon coordinator session — 4,242 events in `08fbfe17`, 1,909 in `c12d...`). Multi-session only when extended via handover paste. Internal "agents per wave" are conceptual roles inside the coordinator loop, NOT separate sessions or Task subagents in the AngelEye sense.                                                                                                                                                               |
| **Aggregation key**      | **`trigger_command =~ /ralphy/` is the only durable signal.** Campaign identity has no on-session field — the campaign concept lives in the worktree path / PR id / BACKLOG entry but isn't extracted. AGENTS.md profile (Dev/Analysis/Content) is inferable from agent count + wave shape but not from any registry field.                                                                                                                                   |
| **Outcomes**             | Wave completed (delivery-review passes). Campaign closed (final assessment.md generated, BACKLOG.md updated, worktree removed via human review gate). KDD auto-promotion (high/critical learnings to permanent KDD).                                                                                                                                                                                                                                          |
| **Linkage signals**      | `trigger_command =~ /^(?:appydave:)?ralphy$/`. Trigger evolved from `/ralphy` to `/appydave:ralphy` mid-stream; both resolve. **Subtype is `build.campaign`** in the registry (not `build.ralphy_campaign` as previously believed — that's only `subtype_heuristic`). Discoverable filter: `subtype === 'build.campaign' AND trigger_command =~ /ralphy/`.                                                                                                    |
| **Internal anatomy**     | Coordinator loop iterates waves. Each wave runs N "agents" (conceptual — same Claude session running specialised passes, not Task subagents). Sample: `27614594` (2026-04-27, heavy), `129a06a1` (2026-04-22, marathon), `01b72d35` (2026-04-20, marathon). The "wave boundary" is observable as a delivery-review pass + AGENTS.md update — would need event-level inspection to reconstruct.                                                                |
| **Corpus evidence**      | **74 Ralphy sessions** across 5+ projects in 6 weeks. Distribution: awb ~24, angeleye ~8, flihub ~8, appydave-plugins ~6, kiros ~3. Average ~12/week. Sample corpus IDs (probed 2026-05-13): `9d1b6bfc` (2026-04-10), `3e649839` (2026-04-07), `27614594` (2026-04-27), `129a06a1` (2026-04-22, marathon), `01b72d35` (2026-04-20, marathon).                                                                                                                 |
| **Open questions**       | (a) Is the _campaign_ the structure, or is the _wave_ the structure? Two natural granularities. (b) How is a multi-session extended campaign linked? Worktree path? PR id? Hand-coded `campaign_id`? None visible at session level today. (c) Profile (Dev/Analysis/Content) is critical metadata but only inferable. Should it be on the structure? (d) BACKLOG.md B### IDs are external to AngelEye — does the structure reference them or remain agnostic? |

**Verdict: needs a structure.** Real archetype with consistent shape. Most often within-session, but the _campaign_ concept transcends a single session — the structure is the right place to represent that. Multi-session campaigns are the harder case.

## 3. Ruflo Swarm

**Codified intent**: `appydave:ruflo` is a **wrapper for install/maintenance**, not an executor. NONE of its 9 operations triggers a swarm. Real swarm execution is a separate code path: one session calls `mcp__ruv-swarm__swarm_init({topology, maxAgents, strategy})` + N parallel `Task({prompt, subagent_type, name, run_in_background:true})` calls + `SendMessage` routing between named workers. Four topologies (`hierarchical`, `mesh`, `ring`, `star`) + `adaptive`. David's anti-drift preset: `hierarchical / maxAgents:8 / strategy:specialized / consensus:raft`.

| Aspect                   | Detail                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Scope**                | **Within-session.** Mode B (Claude Code's `Task` tool) spawns workers as tool calls inside the same session. Workers do NOT become separate sessions in our corpus (Task subagents don't get session JSONLs).                                                                                                                                                                                                                                                                                                                                                                          |
| **Lifecycle model**      | **Topology + N parallel workers.** Not phases (sequential) — topology (graph-shape) + spawn order. Pre-built templates exist (`feature`, `security`, `refactor`, `bugfix`) but most swarms are LLM-composed at runtime from CLAUDE.md instructions.                                                                                                                                                                                                                                                                                                                                    |
| **Constituent sessions** | 1 orchestrator session containing N `Task` tool calls + SendMessage events + post-task hooks. No worker sessions in the registry. Worker identity is in tool args (`subagent_type`, `name`) — not first-class.                                                                                                                                                                                                                                                                                                                                                                         |
| **Aggregation key**      | **No real swarm has fired in the entire corpus (0 instances)**, so the aggregation key is theoretical: `swarm_init.id` (from MCP tool result) + parent `session_id`. Tier-1 detection signal: Bash `~ /claude-flow.*(swarm\|hive\|orchestrate)/` OR MCP tool `mcp__ruv-swarm__swarm_init`. Neither has fired in 1,161 sessions.                                                                                                                                                                                                                                                        |
| **Outcomes**             | Swarm completed (all workers returned, lead synthesised, results posted). Aborted. Topology-specific patterns: hierarchical → queen approves, mesh → consensus reached, ring → pipeline drained.                                                                                                                                                                                                                                                                                                                                                                                       |
| **Linkage signals**      | Today: **zero**. `trigger_command` is `null` on every Ruflo-related session. `subtype_heuristic` may say `build.ruflo_orchestrator` but final subtype is `build.campaign` (same divergence as Ralphy). If a real swarm ever runs: deterministic via MCP tool + Bash signals from `ruflo-topology-research.md:427-433`.                                                                                                                                                                                                                                                                 |
| **Internal anatomy**     | Specified, not observed. Single assistant turn typically contains: 1× `swarm_init` MCP call + N× `Task({run_in_background:true})` + 1× `SendMessage` kick-off + 1× `TodoWrite` shared list. Subagents run parallel; SendMessage between named peers; lead integrates. Then `post-task` hooks fire, `neural_train` + `memory_store` persist patterns.                                                                                                                                                                                                                                   |
| **Corpus evidence**      | **8 ruflo-keyword sessions, 0 real swarm invocations.** All 8 are wrapper-maintenance or research. Sample corpus IDs: `a6fde96f` (2026-05-06, appyctrl, marathon — full install workflow + 17 ordinary Task subagents, NOT swarm workers), `9159d6e8` (2026-05-05, brains, heavy — research), `c408f239` (2026-05-06, appyctrl, heavy — research → built "Rufus" companion). `1b03e83b` (2026-05-04, brains, methodology design). **Critical**: data is THIN to ZERO for real swarm execution.                                                                                         |
| **Open questions**       | (a) Should AngelEye define the structure speculatively (per `ruflo-topology-research.md` Proposal 2) so it's ready when the first swarm runs? Or wait for data? (b) Topology is a first-class structural property — but if it's a "topology + N workers" graph, it doesn't fit a phase/wave model. Different structure family. (c) The 17 Task subagents in `a6fde96f` look like a swarm but aren't — they're ordinary Claude Code Agent Teams. **Detecting the difference matters**: subagent_leg session_kind + `<teammate-message>` markers indicate Claude Agent Teams, NOT Ruflo. |

**Verdict: needs a structure EVENTUALLY, but data is too thin to design from.** This is the cleanest negative result in the survey. Modelling Ruflo today would mean inventing — which is exactly what David warned against. Defer until a real swarm fires; in the meantime keep deterministic detection signals ready (Tier 1 from `ruflo-topology-research.md`).

## 4. Playwright Interaction

**Codified intent**: Several distinct forms, not one. The corpus shows ≥4 patterns:

- **UAT runs** (Taylor RA inside BMAD lifecycle — `bmad-sat` agent uses Playwright automatically per BMAD SKILL.md "Taylor runs Playwright automatically — no human prompt")
- **Screenshot tours** (`/appydave:screenshot-tour` skill — "automates full click-through screenshot documentation")
- **Investigation/auth-debug** ("Can I get you to use Playwright MCP to sign in to the application?" — manual debug sessions)
- **Visual implementation** (Mochaccino-driven UI iteration with Playwright snapshots for feedback)

| Aspect                   | Detail                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Scope**                | **Within-session** for all four forms. UAT is a sub-phase of BMAD (nested inside a BMAD lifecycle structure). Screenshot tour is its own bounded skill invocation. Investigation/visual are ad-hoc.                                                                                                                                                                                                                                                                                                                                                                      |
| **Lifecycle model**      | **Varies by form.** Screenshot tour = sequential route walk (manifest-driven). UAT = test-plan sequence with assertions. Investigation = exploratory tree. Visual impl = iterative compare-and-adjust loop. No common lifecycle shape.                                                                                                                                                                                                                                                                                                                                   |
| **Constituent sessions** | 1 session per form invocation, typically. Tour sessions are often `tool_pattern: playwright-heavy` with 165+ browser actions in one session (per ralphy-synthesis sample `129a06a1`).                                                                                                                                                                                                                                                                                                                                                                                    |
| **Aggregation key**      | Mixed. Screenshot tour: `trigger_command === 'appydave:screenshot-tour'` (exactly 1 hit in corpus — `52c3ca5b`). UAT: no trigger; identified via `has_playwright_calls: true AND parent_bmad_orchestrator`. Investigation: no trigger; identified via `has_playwright_calls: true AND first_prompt mentions auth/debug`. Visual: no trigger; co-occurs with `Mochaccino` keywords.                                                                                                                                                                                       |
| **Outcomes**             | Tour: complete screenshot set written to filesystem + report (`52c3ca5b` produced "detailed report of every screen"). UAT: pass/fail verdict with assertion-level detail. Investigation: hypothesis confirmed/refuted. Visual: design accepted by operator.                                                                                                                                                                                                                                                                                                              |
| **Linkage signals**      | `has_playwright_calls: true` (existing predicate, ingest-time). `tool_pattern: playwright-heavy`. Beyond that, **form discrimination is text-based** — first prompt keywords, presence of route manifest, parent BMAD context. No structured field distinguishes UAT from screenshot tour from investigation.                                                                                                                                                                                                                                                            |
| **Internal anatomy**     | **Each form has its own anatomy**: Tour = (navigate → snapshot → next route) × N. UAT = (login → drive scenario → assert) × N test cases. Investigation = (hypothesis → probe → revise) × N rounds. Visual = (snapshot → compare → adjust code → repeat). No shared internal shape.                                                                                                                                                                                                                                                                                      |
| **Corpus evidence**      | Screenshot tour: only **1** explicit `appydave:screenshot-tour` hit (`52c3ca5b`, 2026-04-09, signal-studio, `tool_pattern: playwright-heavy`, `session_subtype: build.campaign`). Investigation/visual: 9 sessions matched `Playwright MCP` search — sample IDs: `6f007464` (2026-03-24, marathon, auth debug), `3b2ea9c7` (2026-04-13, prod check), `5b6065f9`/`1f23c692`/`95d99e79` (2026-03-18, all `tool_pattern: playwright-heavy`, headed mode requests). UAT within BMAD: implicit in Taylor RA phase of every BMAD story lifecycle (no separate registry trace). |
| **Open questions**       | (a) Is this ONE structure with a `form` discriminator (UAT/tour/investigation/visual), or are these 4 different structures? (b) UAT is intrinsically nested inside a BMAD structure — does it get its own structure, or does it live as a sub-phase of the parent BMAD lifecycle? (c) Should screenshot-tour grow its own subtype rather than `build.campaign`? (d) Investigation form is hard to distinguish from generic debugging that _uses_ Playwright.                                                                                                             |

**Verdict: ambiguous — likely 4 different structures, or one with a strong discriminator.** Most ambiguous candidate in the survey. The "Playwright interaction" framing groups them by tool; the actual lifecycles share little. Pattern: when there's a skill (`screenshot-tour`), structure is clearer. Without a skill, the form is text-detectable but not structurally separable.

## 5. ALS Delamain Batch

**Codified intent**: ALS (Agentic Loop System) delamain workers — a dispatcher spawns N parallel `claude` processes, each in an isolated git worktree at `~/.worktrees/delamain/bulk-analysis/<task>/`, each running the same dispatcher-fed prompt template. The dispatcher orchestrates them externally (not from inside a Claude session). Each worker is a "leg" of the batch.

| Aspect                   | Detail                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Scope**                | **Cross-session, dispatcher-external.** N worker sessions per batch, one per worktree. The dispatcher session that orchestrates them is NOT visible to AngelEye (it's the ALS runtime, not Claude Code).                                                                                                                                                                                                                                                                                                             |
| **Lifecycle model**      | **Parallel fan-out (no phases, no waves).** All workers run the same prompt; no sequential dependencies between them. The batch is "started → all workers in-flight → all workers complete" — closest to fork/join.                                                                                                                                                                                                                                                                                                  |
| **Constituent sessions** | N main-kind sessions, each with `cwd ~ ^.+/\.worktrees/delamain/[^/]+/.+$` and identical first user prompt (e.g. P07_EMOTIONAL_TONE template). 73 such sessions in the 2026-05-07 → 2026-05-13 window alone. Project field today: `d-<hex>` derived from worktree dirname (12+ distinct pseudo-projects polluting the corpus).                                                                                                                                                                                       |
| **Aggregation key**      | **Worktree path + first prompt template + time window.** All visible at session_start. Per the new requirement (`2026-05-13-detect-als-delamain-workers.md`), `project` will be canonicalised to `'als-delamain'`. A `batch_id` would be inferable from "all sessions with the same prompt template started within N minutes of each other" but isn't an existing field — would need a dispatcher-side marker (cwd subdir name like `bulk-analysis/<batch>/` IS the natural key).                                    |
| **Outcomes**             | Each worker produces a deliverable (analysis output written to the worktree). Batch outcome = N deliverables collected. Workers are user-driven-LOOKING but `is_machine_initiated: true` and `is_junk: false` (they produce real output, unlike silent probes).                                                                                                                                                                                                                                                      |
| **Linkage signals**      | `cwd ~ /\.worktrees/delamain/[^/]+/.+/` is deterministic at session_start. Same first user prompt across all workers in a batch (text-equality, ingest-detectable). Same parent `bulk-analysis/<task>/` directory tags them as belonging to the same batch.                                                                                                                                                                                                                                                          |
| **Internal anatomy**     | Each worker is a normal Claude Code session — internally just a single-prompt-and-respond shape. No subagents, no Task tool, no orchestration inside the worker. The structure is entirely _between_ workers (the batch), not within.                                                                                                                                                                                                                                                                                |
| **Corpus evidence**      | **73 worker sessions in 7 days.** Sample IDs from the requirement spec: `21fa9582` (2026-05-11), `5f592574` (2026-05-08). Most others tagged to `d-<hex>` pseudo-projects. All sharing identical P07_EMOTIONAL_TONE first prompt. (Note: at probe time 2026-05-13 the canonicalisation rule hadn't shipped, so `21fa9582` shows `project: app.supportsignal.com.au` — its cwd was a non-delamain location even though the requirement uses it as evidence. The 73-count comes from the requirement spec's analysis.) |
| **Open questions**       | (a) Is the _batch_ the structure, or the _worker_? Each worker is meaningful only as part of a batch. (b) What about partial completion — N-1 workers finish, 1 stalls? Batch is "ended" but incomplete. (c) The dispatcher is external to AngelEye — should batches be self-referential (define themselves from observed workers) or wait for a dispatcher-side hook to register the batch? (d) Different prompt templates (P07, P11, etc.) might be different _kinds_ of batch.                                    |

**Verdict: needs a structure — but it's a fork/join, not a lifecycle.** Clear, repeated, deterministic shape. Importantly: the structure is the BATCH (N workers viewed together), not the worker (one worker has no interesting structure on its own).

## 6. Paperclip Hosting

**Codified intent**: Paperclip is the "company" control plane (REST API on `127.0.0.1:3100`). Each Paperclip workspace hosts a Claude agent. Agents wake on heartbeat; Paperclip spawns `claude` as a subprocess (via `claude_local` adapter), injects `PAPERCLIP_*` env vars, and resumes the same session_id across wake-ups (the same logical agent has a long heartbeat history under one session id). One workspace = many sessions over its lifetime.

| Aspect                   | Detail                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Scope**                | **Cross-session per workspace, app-bounded across lifetime.** A workspace is alive as long as Paperclip is running; it hosts one logical agent that produces many real Claude sessions (or one session resumed many times — see open question).                                                                                                                                                                                                                                                                                                                                                                                              |
| **Lifecycle model**      | **No phases.** Heartbeat-driven. The workspace is the _persistent container_; individual sessions are _instances of work_ within it. Closer to a long-running process model than a lifecycle.                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| **Constituent sessions** | All sessions with `cwd ~ ^.+/\.paperclip/instances/[^/]+/workspaces/<uuid>/` (fallback-workspace runs) OR `cwd == <workspace's configured cwd>` with the deterministic first-prompt template `^-?\nYou are agent <UUID> \(.+\)\. Continue your Paperclip work\.$` (configured-workspace runs). 18 JJ hosted-agent sessions + 3 operator sessions + 8 escape sessions in historic corpus.                                                                                                                                                                                                                                                     |
| **Aggregation key**      | **Workspace UUID** (from cwd or from the agent UUID in first prompt). Visible at session_start. **Workspace UUIDs are not stable IDs** — old AngelEye Factory workspaces (`cfcc0c4b`, `54ea7cf7`, `8fd2ea7b`) are gone, replaced by current JJ UUIDs (`27231022`, `b38ea7f3`). Workspaces can be torn down + replaced.                                                                                                                                                                                                                                                                                                                       |
| **Outcomes**             | Per-heartbeat: a unit of work done (issue progressed, comment posted, agent slept). Per-workspace: indefinite — workspaces exist until the operator deletes them. Per-company: bigger structure outside AngelEye's view.                                                                                                                                                                                                                                                                                                                                                                                                                     |
| **Linkage signals**      | **Three tiers**: (1) prompt template regex (highest confidence, 18/18 hosted hits); (2) workspace cwd regex (catches fallback-workspace runs, 8/8 escape ledger); (3) PAPERCLIP\_\* env vars at session_start (authoritative if Claude Code's hook input exposes them).                                                                                                                                                                                                                                                                                                                                                                      |
| **Internal anatomy**     | Each session is a normal Claude Code session — heartbeat prompt → load Skill → tool-heavy execution. No subagents inside. The structure is entirely cross-session per workspace (and one level up, per company).                                                                                                                                                                                                                                                                                                                                                                                                                             |
| **Corpus evidence**      | **21 historic sessions in local Roamy registry** (live M4 corpus returned 0 paperclip hits at probe time 2026-05-13). Hosted-agent: 18 (15× `27231022` agent UUID, 3× `b38ea7f3`). Operator: 3 (David driving Paperclip — `bb70eaaf`, `7a463d02`, `8ebde1d0`). Escape ledger: 8 workspace-pathed sessions (no longer in registry post-2026-05-07 corpus reset). Live workspace dirs today: 5 distinct UUIDs at `~/.paperclip/instances/default/workspaces/`.                                                                                                                                                                                 |
| **Open questions**       | (a) Is a Paperclip _workspace_ the structure, or a Paperclip _company_ (containing N workspaces)? Or a Paperclip _heartbeat run_ (one wake-up)? Three natural granularities. (b) `--resume`-driven session_id reuse: are 100 heartbeats one session or N sessions? Paperclip's `claude_local` adapter resumes; AngelEye sees them as one session JSONL growing over time. (c) Workspace UUID is not stable — can't be a structure primary key on its own. Needs company id + workspace label + UUID + creation time. (d) Configured vs fallback workspace are two states of the same workspace — does the structure track state transitions? |

**Verdict: needs a structure.** Distinct, multi-session, app-driven shape. Workspace is the natural granularity. The "platform-driven origin" (Paperclip starts the work, not David) makes this categorically different from BMAD/Ralphy/Ruflo where Claude Code is the initiator.

## 7. AppyCtrl Probe Sequence

**Codified intent**: AppyCtrl (T3 Code fork) runs a `probeClaudeCapabilities` call every 5 minutes per Claude provider instance. Each probe spawns `claude` as a subprocess, reads `initializationResult()` (account info + slash commands), and aborts the subprocess WITHOUT yielding any user prompt. Result: a session with `session_start` + N `instructions_loaded` + `session_end` events, **zero `user_prompt` events**. The session exists only because the SDK API surface requires it.

| Aspect                   | Detail                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Scope**                | **Each probe is its own session.** The "sequence" is the temporal pattern of probes — one every 5 minutes, per provider instance, while AppyCtrl is running. App-bounded.                                                                                                                                                                                                                                                                                                 |
| **Lifecycle model**      | **Stateless aggregate.** No phases, no waves, no internal structure. The probe is atomic (single SDK call). The _sequence_ is just a heartbeat — at-cadence, at-volume.                                                                                                                                                                                                                                                                                                   |
| **Constituent sessions** | One session per probe. Currently 1,007 silent sessions corpus-wide (651 tagged to `appyctrl`, 320 to `app.supportsignal.com.au`, etc — the project distribution is just a leak of what tab the operator had open). At 5-min cadence: 288 probes/day per provider instance.                                                                                                                                                                                                |
| **Aggregation key**      | **None at session level today.** The "sequence" is implicit in `session_class === 'machine_signal' AND project === 'appyctrl' AND last_active timestamps form a 5-min cadence`. No structured `probe_run_id` exists. AppyCtrl itself doesn't issue a batch id — each probe is independent.                                                                                                                                                                                |
| **Outcomes**             | Per probe: capability metadata returned (success) or subprocess crash (failure). No deliverable. **Probes correctly tagged `is_junk: true`** per the silent-session filter. Sequence-level outcome would be uptime / heartbeat health ("did AppyCtrl probe in the last 5 minutes?").                                                                                                                                                                                      |
| **Linkage signals**      | `session_class: 'machine_signal'` (deterministic — `meta.silent_session` subtype + `is_junk: true`). At probe time 2026-05-13: 195/200 sampled machine_signal sessions are project=appyctrl, 100% with cwd inside `/Users/davidcruwys/dev/ad/apps/appyctrl` (or the operator's currently-open tab).                                                                                                                                                                       |
| **Internal anatomy**     | None. Each probe has ~7 hook events (session_start + N instructions_loaded + session_end), zero user prompts, zero tool uses, subprocess aborted before any conversation happens. The session is structurally a _placeholder_ — its existence is the signal.                                                                                                                                                                                                              |
| **Corpus evidence**      | **1,007 silent sessions corpus-wide.** Sample probed 2026-05-13: 195 of 200 machine_signal sessions are AppyCtrl probes (in `/Users/davidcruwys/dev/ad/apps/appyctrl`). Cadence confirmed via timestamps spaced ~5 min apart (per appyctrl-synthesis evidence: `03:01:02 → 02:55:54 → 02:50:48 → 02:45:41 → 02:40:33` etc — five-minute intervals dead-on, matching `SNAPSHOT_REFRESH_INTERVAL = Duration.minutes(5)`).                                                   |
| **Open questions**       | (a) **Does this need a structure at all?** A probe is atomic; the "sequence" is just a cadence pattern, not a lifecycle. (b) If it does: what does "an AppyCtrl probe sequence" represent? The day's probes? The app-running interval? The provider instance's lifetime? (c) Should this be answered at the _tag layer_ (each probe gets `tag: appyctrl_probe`) and the _aggregate question_ ("is AppyCtrl alive?") handled by a dreaming-pass query, NOT by a structure? |

**Verdict: probably DOESN'T need a structure.** Strong case for "the tag layer is enough." Individual probes are atomic and structureless. The aggregate question ("is AppyCtrl alive? what was the recent probe cadence?") can be answered by querying tagged sessions over a time window — it doesn't need a persisted structure entity. **The aggregate is the answer to a query, not an object.**

## Comparative Section

### 1. Shared shape across all candidates

Fields that appear (in some form) in every candidate's natural representation:

| Field        | Notes / variants                                                                                                                                                                                                                       |
| ------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `id`         | Stable identifier for the structure instance                                                                                                                                                                                           |
| `kind`       | Discriminator: `bmad_lifecycle` / `ralphy_campaign` / `ruflo_swarm` / `playwright_interaction` / `als_delamain_batch` / `paperclip_workspace` / `appyctrl_probe_sequence`                                                              |
| `started_at` | First constituent event/session                                                                                                                                                                                                        |
| `ended_at`   | Last constituent event/session — **nullable** for app-bounded structures (Paperclip workspace lifetime, AppyCtrl-while-running)                                                                                                        |
| `sessions[]` | Constituent session IDs. Nullable/empty for within-session structures where the "container" IS the session itself (Ralphy single-session, Ruflo swarm) — for those, `sessions[]` is `[parent_session_id]` and the work lives in events |
| `outcome`    | Free-form or enum — varies. **Doesn't apply uniformly to AppyCtrl** (probes are atomic, no outcome beyond success/fail; "sequence outcome" is undefined)                                                                               |
| `project`    | The project context — derived from constituent session cwds. Always meaningful EXCEPT delamain (where `project` is the canonicalised `als-delamain`, not the real work-domain project)                                                 |
| `confidence` | How sure are we this structure exists? Useful when aggregating from heuristics                                                                                                                                                         |

**Fields that DON'T generalise (candidate-specific even though some intuitively shared)**:

- **`phases[]`** — only BMAD has true named ordered phases. Ralphy has waves (similar but not identical — fixed wave size by profile, repeating shape). Ruflo has topology+workers. Delamain has fan-out parallel legs. Paperclip has no phases. AppyCtrl has no phases. → Phases are a BMAD-specific concept.
- **`outcome`** — meaningful for BMAD (story shipped/aborted), Ralphy (campaign closed), Ruflo (swarm completed), Delamain (batch finished), Playwright (test passed / tour complete). **Not applicable** to Paperclip-workspace (indefinite lifetime) and AppyCtrl-probe-sequence (no outcome beyond per-probe success). → Some structures are ongoing, not "completed".
- **`topology`** — only Ruflo has it.
- **`waves[]`** / **`profile`** — only Ralphy.
- **`workspace_uuid`** / **`agent_uuid`** / **`company_id`** — only Paperclip.
- **`worktree_path`** / **`prompt_template`** / **`batch_id`** — only Delamain.

### 3. Patterns of divergence — three structural pattern families

Grouping the divergent shapes reveals **three distinct structural pattern families** (David's "shared structural patterns"):

#### Family A — Cross-session phased lifecycle ("sequential pipeline with named phases")

- **BMAD** (Mode 2 era — pre-2026-04-06)
- **Ralphy** when extended across handovers
- Conceptually: **Playwright UAT** as a sub-phase of BMAD

Shared shape: ordered `phases[]`, each phase has a role/identity, transitions with optional gates, fix-loops, outcomes per phase that roll up. **Cross-session aggregation key in `trigger_arguments` (story_id) or domain id (campaign_id).**

#### Family B — Within-session orchestrated workflow ("one session hosting an internal pipeline or topology")

- **BMAD** (Mode 1 era — current default; in-process Agent Teams)
- **Ralphy** (typical case — single coordinator loop)
- **Ruflo** (within-session swarm via Task tool)
- **Playwright screenshot-tour** (one skill, one bounded sequence)
- **Playwright investigation/visual** (one ad-hoc loop)

Shared shape: 1 parent session; internal anatomy described by event sequence (tool calls, sub-agent invocations, SendMessage routing, TodoWrite items). **No `sessions[]` plural — just one `parent_session_id` + internal event ranges.** The structure is event-level, not session-level.

#### Family C — Persistent/cyclic external-driven aggregate ("N independent units, no internal lifecycle, exists while host is running")

- **Paperclip workspace** (N heartbeat sessions over indefinite lifetime)
- **Delamain batch** (N parallel worker sessions, one-shot fan/join)
- **AppyCtrl probe sequence** (N probes at fixed cadence) — _if_ it gets a structure at all

Shared shape: N constituent sessions, **no internal lifecycle on the structure itself**, identified by an external aggregator (workspace UUID / batch dispatcher id / probe cadence). The structure is an _aggregate over sessions_, not a workflow.

#### Asymmetries inside the families

- **Within Family A**: BMAD phases are named and required-ordered; Ralphy waves are numbered and same-shape-repeating. Different at the schema level.
- **Within Family B**: BMAD-Mode-1 vs Ruflo-swarm are structurally similar (one session, N internal agent invocations) but topology is fundamentally different — BMAD is a pipeline, Ruflo can be a graph. So even within Family B there's a sub-distinction between _internal pipeline_ and _internal topology_.
- **Within Family C**: Paperclip (long-lived, sparse heartbeats), Delamain (short, dense parallel burst), AppyCtrl (continuous fixed-cadence cycle) — three temporal shapes of "aggregate over sessions." Different at the query level even if the schema is the same.

### Implication for the structure-layer design

The "single generic abstraction" approach would force all 7 onto one shape. The "one schema per candidate" approach is 7 separate models with no shared API. The data suggests a **middle path: a small shared base (id, kind, started_at, ended_at, sessions[], confidence) + a per-family variant type carrying family-specific anatomy**.

That gives 3 variant types (A: phased_lifecycle, B: orchestrated_workflow, C: persistent_aggregate) — not 7. Per-kind discriminator fields live inside the family variant. AppyCtrl's "maybe no structure" answer is supportable: it falls out of Family C if added; it can also remain a pure tag-layer concern with no structure at all.

This is the comparative finding to test against the next design round — not a schema, but a strong hypothesis about where the natural lines fall.
